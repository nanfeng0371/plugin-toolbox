## 一、背景与目标

### 1.1 背景

调课助手 Chrome 扩展（V3）已具备批量改约、自然语言解析、学员信息簿、历史日志等核心功能，但存在以下局限：

- **使用门槛高**：需要安装 Chrome 扩展，一线辅导伙伴普遍不会操作
- **仅限桌面**：不支持手机端，移动场景完全缺失
- **依赖千问浏览器**：绑定特定浏览器环境，换设备即失效
- **共享困难**：无法在团队成员间共享学情表、调课记录

### 1.2 目标

开发调课助手网页版（Web App），满足以下核心目标：

1. **零门槛**：一线辅导伙伴打开网页即可使用，无需安装任何插件
2. **移动友好**：支持手机端，在企微、微信内置浏览器、Safari/Chrome 均可正常使用
3. **团队共享**：学情表一次上传，全团队共享；调课记录可追溯
4. **自动 Token**：配合现有 Chrome 扩展一键同步，不需要伙伴手动复制 JWT

### 1.3 成功指标

- 一线辅导伙伴从打开网页到完成第一次调课，操作步骤 ≤ 5 步
- Token 同步操作"点一个按钮"即可完成，无需了解技术细节
- 批量调课 10 条以内，全流程 30 秒内完成

## 二、用户分析

### 2.1 用户角色

| 角色 | 使用场景 | 技术水平 | 核心诉求 |
|------|---------|---------|---------|
| **辅导伙伴（counselor）** | 日常改约、应急调课 | 零技术基础 | 快、简单、不出错 |
| **年级小组长（gradeLeader）** | 批量调课、查看记录 | 低技术基础 | 批量操作、可回溯 |
| **学科负责人（subjectLeader）** | 统筹调度、监控执行 | 中等技术 | 批量+状态总览 |
| **运营负责人（operationLeader）** | 管理学情表、查看所有记录 | 较高技术 | 管理权限、完整历史 |

### 2.2 核心用户画像

**主要用户：辅导伙伴**
- 每天需要处理 1-10 次调课请求
- 日常工具是企微，不常打开电脑
- 完全不懂技术，看到"F12""JWT""Cookie"就放弃
- 习惯用手机操作，文字输入接受度高

### 2.3 访问权限

**准入条件**：拥有 EduFlow 系统账号，角色为 counselor（辅导伙伴）及以上。

即：凡是能登录 EduFlow 消息平台的成员，均可使用调课助手网页版。调课助手内部不做角色区分，所有登录用户的功能权限相同。

### 3.2 P0 核心功能（MVP 必须上线）

#### F01：EduFlow 账号登录

**描述**：用户使用 EduFlow 系统的账号密码登录调课助手。

**流程**：
1. 打开网页 → 显示登录页
2. 输入用户名 + 密码 → 点"登录"
3. 后端调用 ef-api `auth.login` 接口验证
4. 成功 → 签发 ef_token，写入 localStorage，跳转主页
5. 失败 → 提示"账号或密码错误"

**字段**：
- 用户名：文本输入框
- 密码：密码输入框
- 登录按钮（大尺寸，手指友好）

**记住状态**：ef_token 有效期 2 小时，到期自动跳转登录页（与 EduFlow 行为一致）

---

#### F02：调课 JWT Token 管理

**背景**：调课助手需要调用 ai-genesis.yuaiweiwu.com 的内部 API，这些 API 需要 JWT（来自钉钉 CAS SSO），24小时有效期。

**痛点**：普通伙伴不会从 F12/DevTools 复制 JWT，必须设计一键操作。

**设计方案：引导跳转 + 手动复制粘贴 + 剪贴板自动检测**

> **设计约束**：用户完全依赖手机操作，无电脑可用。Chrome 扩展方案不适用。

流程：
```
伙伴每天操作流程（一次，24小时内有效）：
1. 在调课助手网页上点击"获取 Token"按钮
2. 新标签页打开 ai-genesis 后台（企微自动 SSO 登录）
3. 从 ai-genesis 复制 Token（长按地址栏或复制按钮）
4. 切回调课助手 → 自动检测剪贴板 JWT 格式 → 自动填充
5. 点击"确认保存" → Token 状态变为"有效"
```

网页端 Token 获取界面：
- 大按钮"🔑 获取 Token"，点击后新标签页打开 ai-genesis
- 粘贴区域（自动聚焦，支持长按粘贴）
- 剪贴板自动检测：切回页面时自动读取剪贴板，识别 JWT 格式（`eyJ` 开头，三段式）自动填充
- 验证按钮：粘贴后点"验证并保存"，后端验证 JWT 有效性 + 解码获取过期时间
- 保存：AES 加密存储到 CloudBase `tk_tokens`

网页端 Token 状态展示：
- ✅ 有效（剩余 XX 小时）
- ⚠️ 即将过期（剩余 <2小时，橙色提示）
- ❌ 已过期 / 未设置（红色，显示获取步骤引导）

**未设置 Token 时的引导文案**：
> "调课 Token 未设置，无法执行调课操作。
> 请按以下步骤操作：
> 1. 点击下方「获取 Token」按钮
> 2. 在新页面中登录调课后台（企微会自动登录）
> 3. 复制页面中的 Token
> 4. 切回本页面，Token 会自动填入
> 5. 点击「确认保存」即可"

**Token 粘贴区设计**：
- 单行文本框，placeholder："粘贴 Token（eyJ...）"
- 右侧"粘贴"按钮（移动端友好）
- 自动检测：`navigator.clipboard.readText()` 或页面 `visibilitychange` 事件触发剪贴板读取
- JWT 格式校验：`^eyJ[A-Za-z0-9_-]+\.eyJ[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+$`
- 校验通过后自动触发验证保存

#### F03：调课指令输入与解析

**描述**：用户输入调课指令（支持自然语言和结构化格式），系统解析为标准化调课任务列表并预览确认。

**输入格式支持**：

*格式一：结构化（4列，Tab/空格/逗号分隔）*
```
学员标识  第几讲  新日期  新开始时间
王一      5      5月2日  10:00
12345     3      5-3     14:00
```
- 学员标识支持：学员ID（数字）或姓名（需学情表）
- 日期格式：`MM月DD日`、`MM-DD`、`MM/DD`、`YYYY-MM-DD`
- 时间格式：`HH:mm`、`HH:mm:ss`、`上午10点`、`下午3点`

*格式二：自然语言（每行一条）*
```
王一，第5讲，调到5月2日早上10点上课
把李二的第3讲改到5月3日下午2点
```

*混合输入*：同一文本框内两种格式可以混用，系统自动识别。

**解析预览**：
- 点击"解析预览"按钮，在输入框下方显示解析结果列表
- 每行显示：学员名（或ID）→ 第X讲 → 新时间 → 状态（已解析/解析失败+原因）
- 解析失败的行高亮红色，用户可在原始输入里修改后重新解析
- 显示总计：X条成功，Y条失败

**批量执行**：
- 点击"执行调课"按钮，对已解析成功的条目批量执行
- 并发池：3个并发，组内500ms错开，批次间300ms（复用 V3 扩展的逻辑）
- 执行过程实时显示进度：第X条/共Y条
- 每条执行结果：✅ 成功 / ❌ 失败（带原因）

#### F04：学情表管理（双模式支持）

**背景**：一线伙伴更习惯用学员姓名，不记得学员ID；但系统API需要的是学员ID。学情表是姓名→ID的桥梁。

**双模式设计**：

| 模式 | 使用方式 | 适用场景 |
|------|---------|---------|
| 直接ID模式 | 输入里直接写学员ID | 熟悉系统、快速操作 |
| 姓名匹配模式 | 输入姓名，系统查学情表→找ID | 一线伙伴日常使用 |

两种模式自动切换：系统识别输入是数字→走ID模式，识别是姓名→走姓名匹配模式。

**学情表功能**：

上传：
- 设置页 → 学情表 → 上传 Excel/CSV 文件
- 支持格式：`.xlsx`、`.xls`、`.csv`
- 必填列：姓名、学员ID
- 可选列：手机号（用于重名消歧）
- 上传成功提示：共导入 X 条学员信息

存储：
- 存在 CloudBase `tk_students` 集合（该账号下的学情表，跨设备可用）
- 每次上传**覆盖**旧数据（不做增量合并，简单可靠）

姓名匹配逻辑（复用 V3 扩展的 `matchStudent`）：
1. 精确姓名匹配（唯一结果）→ 直接使用
2. 精确姓名匹配（多个结果/重名）→ 提示"存在同名学员，请补充手机号后4位"
3. 模糊匹配（名字包含关系）→ 显示候选列表，让用户确认
4. 完全未匹配 → 报错"未在学情表中找到XXX"

---

#### F05：历史调课日志

**描述**：每次执行的调课操作自动记录，支持按日期查看和搜索。

**记录内容**：
- 执行时间
- 操作人（当前登录用户）
- 调课详情：学员名/ID、第X讲、原时间（如有）→ 新时间
- 执行结果：成功/失败（失败原因）

**展示**：
- 按日期分组，最新在前
- 每组显示日期标题 + 当天的调课记录
- 每条记录点击可展开详情

**存储**：CloudBase `tk_logs` 集合，按用户隔离（只能看自己的记录），最多保留 1000 条

**搜索**：按学员姓名/ID搜索历史记录

## 四、非功能需求

### 4.1 兼容性

| 环境 | 要求 |
|------|------|
| iOS Safari | 完全支持（iPhone 主力场景） |
| Android Chrome | 完全支持 |
| 企微内置浏览器 | 完全支持（重点场景） |
| 桌面 Chrome/Edge | 完全支持 |
| 千问浏览器（桌面） | 完全支持 |

### 4.2 性能

- 页面首次加载 < 2 秒（4G 网络）
- 批量调课 10 条，全部执行完成 < 15 秒
- 课表查询（单个学员）< 3 秒

### 4.3 安全

- 调课 JWT 不在前端持久化，仅在云函数内使用
- AES 加密存储于 CloudBase
- ef_token 2小时有效期，到期自动登出
- 所有 tk-api 请求必须携带 ef_token

### 4.4 可用性

- 无操作步骤超过 3 层（不需要深度导航）
- 错误提示使用人话，不展示技术术语
- 所有按钮最小触控尺寸 44×44px

## 六、技术路径

### 6.1 系统架构

```
用户设备（手机/电脑浏览器）
      │
      │  HTTPS
      ▼
CloudBase 静态托管 /tk/
（HTML + CSS + JS，纯静态）
      │
      │  POST /tk-api（CloudBase 网关）
      ▼
CloudBase 云函数 tk-api
（Node.js 18，Event 类型，与 ef-api 相同模式）
      │
      ├──→ CloudBase NoSQL（ef_users/ef_tokens/tk_tokens/tk_students/tk_logs）
      │
      └──→ ai-genesis.yuaiweiwu.com（调课 API）
             Cookie: authorization-app=aiXin; authorization-token=<JWT>
```

### 6.4 API 路由设计（tk-api）

```
POST /tk-api
Body: { "action": "<service>.<method>", "data": {...}, "_token": "<ef_token>" }

公开接口（无需 ef_token）：
  auth.login            ← 用 EduFlow 账户登录

需 ef_token 的接口：
  auth.verify           ← 验证 ef_token
  
  tiaokeToken.save      ← 保存/更新调课 JWT（来自扩展同步）
  tiaokeToken.get       ← 获取当前调课 JWT 状态
  
  student.import        ← 上传/覆盖学情表（Excel 解析在前端完成，传数组）
  student.list          ← 获取学情表（用于前端预览，最多100条）
  student.clear         ← 清空学情表
  student.match         ← 按姓名匹配学员 ID
  
  class.list            ← 查询学员课表（透传 ai-genesis）
  class.getHour         ← 查询课时详情（透传 ai-genesis）
  
  reschedule.parse      ← 解析调课指令（不执行，返回任务列表）
  reschedule.execute    ← 执行单条改约
  reschedule.batch      ← 批量执行（并发池 3）
  
  log.list              ← 查询历史调课记录
  log.search            ← 按关键词搜索历史记录
```

### 6.5 数据模型

**tk_tokens 集合**
```json
{
  "_id": "<userId>",           // 以 userId 为主键，一人一条
  "userId": "xxx",
  "encryptedToken": "xxx",     // AES 加密后的调课 JWT
  "expireTime": 1234567890,    // JWT 过期时间戳（从 JWT 解码）
  "updatedAt": "2026-05-25T10:00:00Z"
}
```

**tk_students 集合**
```json
{
  "_id": "auto",
  "ownerId": "<userId>",       // 上传人，按用户隔离
  "name": "王一",
  "studentId": "12345",
  "phone": "13812345678",      // 可选，用于重名消歧
  "updatedAt": "2026-05-25"
}
```

**tk_logs 集合**
```json
{
```

### 6.6 Chrome 扩展改动（V3 新增功能）

在 `popup/popup.html` 和 `popup/popup.js` 中新增"同步 Token"功能区：

```
新增 UI 区域：
┌───────────────────────────┐
│ 🔗 调课助手网页版          │
│ ─────────────────────── │
│ 网页端 Token：未设置 ❌   │
│ [▶ 一键同步 Token]        │
└───────────────────────────┘

逻辑：
1. background.js 新增 action: "syncTiaokeToken"
2. 读取当前页 Cookie: authorization-token（通过 chrome.cookies.get）
3. 调用 tk-api tiaokeToken.save，需要知道当前用户的 ef_token
   → ef_token 存在 popup 的 chrome.storage.local 里（V3 现有逻辑）
4. 显示同步结果
```

### 6.7 关键技术决策

| 决策点 | 选择 | 原因 |
|--------|------|------|
| Excel 解析位置 | **前端**（SheetJS）| 省流量，复用 V3 扩展已有的 SheetJS 逻辑 |
| 调课 JWT 存储 | 云端加密（AES）| 前端不持久化 JWT，安全边界清晰 |
| 用户体系 | 复用 EduFlow ef_users | 不另建系统，零运维成本 |
| 学情表归属 | 按用户隔离（ownerId） | 防止伙伴误看他人学情 |
| 打包方式 | esbuild IIFE | 绕过 CloudBase CDN ES Module 限制（EduFlow 同款方案） |
| 路由方式 | Hash 路由（#/path） | 静态托管无服务端路由支持 |
| 前端框架 | 无框架（原生 HTML+CSS+JS）| 与 EduFlow 对齐，零构建复杂度 |

## 八、风险与约束

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| 调课 JWT 24小时过期 | 未同步 Token 时无法调课 | Token 状态醒目展示，到期前 2h 提醒 |
| ai-genesis API 不稳定 | 改约失败 | 错误信息透传给用户，支持重试 |
| 学情表姓名重名 | 无法确定目标学员 | 要求补充手机号后4位消歧 |
| 企微内置浏览器兼容性 | 部分功能异常 | 用原生 HTML，避免使用浏览器实验性特性 |
| CloudBase 冷启动 | 首次调用 >3s | 云函数无最大实例限制，接受首次冷启动 |
